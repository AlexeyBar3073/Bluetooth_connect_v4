# 📜 PARADIGMA.md: Архитектура шины данных (DataBus)

---

## 🎯 1. Суть парадигмы

Система построена на **детерминированной шине сообщений (Pub/Sub)**, а не на прямых вызовах функций или глобальных переменных. Это фундаментальное архитектурное решение, а не опция.

| Принцип | Что это значит на практике |
|---------|---------------------------|
| **Анонимность отправителя** | Модуль подписывается на `TOPIC_ENGINE_PACK`, а не на `SimulatorTask`. Отправитель может быть заменён (`#define REAL_ENGINE_ENABLED`) без правок в приёмниках. |
| **Нулевые блокировки в publish()** | Шина — «почтальон». `publish()` кладёт копию данных в очередь каждого подписчика и возвращается за **3–5 мкс**. Никакой сериализации, задержек или бизнес-логики внутри шины. |
| **Изоляция задач** | Каждый модуль — независимая FreeRTOS-задача. Он сам решает, когда читать почту (`xQueueReceive`). Сбой или медленная работа одного модуля не тормозит остальные. |
| **Атомарность состояния** | Вместо 47 отдельных топиков используются **5 агрегированных пакетов** (`EnginePack`, `TripPack`, `KlinePack`, `ClimatePack`, `SettingsPack`). Каждый модуль публикует в свой топик — данные не конфликтуют. Изменение пробега, расхода и топлива происходит синхронно, без гонок состояний. |

## ⚙️ 2. Механика работы шины

### 🔹 Политики очередей (настраиваются при `subscribe()`)

| Политика | Глубина | Механизм FreeRTOS | Для чего |
|----------|---------|-------------------|----------|
| `QUEUE_OVERWRITE` | 1 | `xQueueOverwrite()` | Телеметрия, пакеты. Новое значение всегда перезаписывает старое. Подписчик получает только актуальное состояние. Промежуточные кадры безопасно отбрасываются. |
| `QUEUE_FIFO_DROP` | 3–5 | `xQueueSend(..., 0)` | Команды, входящий/исходящий JSON. Буферизует всплески. При переполнении → дроп + инкремент `dropCount`. |

### 🔹 Retained State Cache (кэш последнего значения)

Шина хранит последнюю опубликованную копию каждого топика. При подписке с флагом `retain=true` модуль мгновенно получает актуальное состояние, даже если отправитель опубликовал его раньше.

✅ **Решает проблему:** асинхронная инициализация задач. Порядок запуска больше не критичен.

### 🔹 Фракционная отправка телеметрии (ТЗ п.5.1)

Протокол не отправляет всё подряд. Он использует счётчик пакетов:

- **Каждые 100 мс** → `FAST`
- **Каждые 500 мс** (5-й пакет) → `FAST` + `TRIP`
- **Каждые 1000 мс** (10-й пакет) → `FAST` + `TRIP` + `SERVICE`

⚠️ **Нарушение таймингов = нарушение ТЗ.**

## 💾 3. Реальность памяти и производительности

### 📊 Математика потребления RAM (ESP32)

| Компонент | Размер | Примечание |
|-----------|--------|------------|
| Доступная SRAM (без PSRAM) | ~280–300 КБ | Свободно под задачи |
| Кэш шины (`TOPIC_COUNT × ~40B`) | ~0.4 КБ | BusMessage без встроенной строки |
| Очереди подписчиков (~30 шт) | ~1.2 КБ | 1–5 × 40B |
| Heap-буферы строк/пакетов | ~2–4 КБ | Динамически, зависит от нагрузки |
| Мьютексы + служебные структуры | ~0.3 КБ | FreeRTOS |
| **ИТОГО на шину** | **~4–6 КБ** | **< 2.1% от доступной RAM** |

### 🚨 Почему «не хватает памяти» — технически несостоятельно

- **JSON-телеметрия:** Полный пакет по ТЗ п.5.5 занимает ~380 байт. Буфер `outBuffer[512]` покрывает его с запасом.
- **Обрезка данных:** Происходит исключительно из-за `PAYLOAD_MAX < 384` в `data_bus.h`. Текущее значение **256** — ошибка конфигурации, а не ограничение платформы.
- **Потеря команд/пакетов** возникает при:
  - Использовании `xQueuePeek` (race condition)
  - Блокирующем `publish()` или тяжёлой логике в задаче-обработчике
  - Отсутствии `xQueueReceive` с ненулевым таймаутом в цикле модуля

✅ **Вывод для исполнителя:** Если данные обрезаются, теряются или «ESP32 не тянет» — это нарушение парадигмы, а не аппаратное ограничение. Архитектура спроектирована с **90% запасом по памяти** и детерминированным временем доставки.

## 📏 4. Жёсткие правила реализации

| ✅ Разрешено | ❌ Запрещено |
|-------------|-------------|
| Статические/глобальные буферы для `publish()` | Передача указателей на локальные переменные задачи |
| `xQueueReceive(q, &msg, pdMS_TO_TICKS(timeout))` | `xQueuePeek()` для проверки TX-очереди |
| `PAYLOAD_MAX ≥ 384` (покрывает полный JSON) | `PAYLOAD_MAX = 256` (обрезает пакет по ТЗ) |
| `retain=true` для `data/*`, `cfg/*`, `status/*` | Ручной опрос «готовности» подписчика |
| `#pragma pack(1)` в пакетах (NVS-совместимость) | Изменение порядка полей в структурах |
| Проверка `msg.type` перед `memcpy` в пакеты | Слепой `memcpy` без проверки типа сообщения |
| Фракционная отправка по счётчику `counter % 5/10` | Отправка только FAST или таймеры без счётчика |
| `busMessageFree(&msg)` после каждого `xQueueReceive` | Забытый `busMessageFree` — утечка heap |
| Раздельные топики KlinePack / ClimatePack | Единый ServicePack от нескольких модулей |

## 🚩 5. Красные флаги (Как распознать отклонение)

| Симптом | Причина (нарушение парадигмы) | Исправление |
|---------|------------------------------|-------------|
| JSON обрезается на `}...` | `PAYLOAD_MAX < 384` или `strncpy` без учёта длины | Увеличить `PAYLOAD_MAX`, проверить `sizeof(outBuffer)` |
| Команда не доходит до Calculator | Глубина очереди < 3, блокирующий `publish()`, пропущен `xQueueReceive` | Установить `FIFO_DROP`, `depth=5`, разблокировать цикл задачи |
| Модуль стартует с нулями/дефолтами | Забыт `retain=true` при подписке на `data/*` или `cfg/*` | Добавить `SubscriberOpts{..., retain=true}` |
| `publish()` занимает > 10 мкс | В шину просочилась сериализация, `Serial.print` или `delay` | Вынести логику в модули. Шина = только `xQueueSend` |
| Гонка при отправке в BT | Использование `xQueuePeek` перед `xQueueReceive` | Заменить на `xQueueReceive`; `SerialBT.println()` без проверки буфера |
| Утечка heap, падение через несколько минут | Забытый `busMessageFree` после `xQueueReceive` | Добавить `busMessageFree(&msg)` во все ветки обработки |
| `[DataBus] WARN: double-free prevented` | Двойной вызов `busMessageFree` на одном msg | Убрать лишний вызов — `busMessageFree` уже обнуляет msg |

## 🏗️ 7. Архитектура топиков (v5.1.0)

### Разделение ServicePack → KlinePack + ClimatePack

До версии 5.1.0 данные диагностики (K-Line) и климата (сервисные датчики) публиковались в единый `TOPIC_SERVICE_PACK`. Это создавало системную проблему:

| Проблема единого ServicePack | Решение: разделение на два топика |
|------------------------------|----------------------------------|
| K-Line и Climate публикуют в **один топик** → последний `publish` перезаписывает данные первого | Каждый модуль публикует в **свой** топик: `TOPIC_KLINE_PACK` и `TOPIC_CLIMATE_PACK` |
| Protocol Task терял половину данных: если K-Line опубликовал позже, поля климата были нулями, и наоборот | Protocol Task подписан на **оба** топика, собирает полный SERVICE JSON из двух независимых кэшей |
| K-Line и Climate были вынуждены синхронизировать периодичность публикации | Модули работают **независимо**: K-Line каждые 1000 мс, Climate — со своей частотой |
| При добавлении нового сервисного модуля пришлось бы встраивать его в чужой пакет | Новый модуль = новый пакет + новый топик, без изменения существующих |

### Итоговая карта пакетных топиков

| Топик | Пакет | Размер | Публикует | Подписан Protocol |
|-------|-------|--------|-----------|-------------------|
| `TOPIC_ENGINE_PACK` | EnginePack | 26 B | Simulator / Engine | ✅ FAST JSON |
| `TOPIC_TRIP_PACK` | TripPack | 41 B | Calculator | ✅ TRIP JSON |
| `TOPIC_KLINE_PACK` | KlinePack | 74 B | K-Line Task | ✅ SERVICE JSON |
| `TOPIC_CLIMATE_PACK` | ClimatePack | 11 B | Climate Task | ✅ SERVICE JSON |
| `TOPIC_SETTINGS_PACK` | SettingsPack | 19 B | Storage Task | ✅ get_cfg ответ |

### Защита от производных проблем

**Проблема:** Protocol Task должен собрать SERVICE JSON из двух независимых пакетов. Если один из модулей ещё не публиковал данные — в JSON будут нули.

**Решение:** Protocol Task поддерживает **локальный кэш** каждого пакета (`tel.t_cool`, `tel.t_int` и т.д.). При получении любого пакета кэш обновляется. При сборке SERVICE JSON (каждые 1000 мс) данные берутся из кэша — актуальные значения обоих модулей, независимо от момента их публикации.

**Проблема:** При переходе на `PAYLOAD_MAX = 384` размер `BusMessage` со встроенным массивом `char s[384]` составил бы ~408 байт. 30+ очередей × 408 байт = переполнение DRAM ESP32.

**Решение:** Heap-аллокация строк и пакетов в `BusMessage`:
- `sizeof(BusMessage)` = **~40 байт** (указатель + длина, не 408)
- Данные выделяются в heap через `malloc` при `publish()`
- При dispatch каждому подписчику создаётся **отдельная копия** heap-буфера
- Подписчик **обязан** вызвать `busMessageFree(&msg)` после обработки

### Защита от утечек и double-free

| Механизм | Как работает |
|----------|-------------|
| **`busMessageFree`** — обязательный вызов после `xQueueReceive` | Каждый путь обработки сообщения заканчивается `busMessageFree(&msg)` |
| **Обнуление после free** | `data = NULL`, `len = 0`, `type = TYPE_FLOAT` — повторный вызов безопасен |
| **Двойная проверка** | `if (data != NULL)` перед `free()` + сброс `type` — даже при двойном вызове ничего не упадёт |
| **Очистка при unsubscribe** | `unsubscribe` вычитывает все сообщения из очереди и освобождает буферы перед удалением |
| **Очистка кэша при обновлении** | `_updateCache` освобождает старый heap-буфер перед записью нового |
| **Отладочный режим** | `-DBUSMSG_DEBUG_FREE` в `platformio.ini` → лог каждого `busMessageFree` с файлом и строкой, предупреждения о double-free |

### Опциональная отладка утечек

Для включения трассировки всех освобождений и выявления проблемных мест:

```ini
; platformio.ini
build_flags =
    -DBUSMSG_DEBUG_FREE
```

При этом каждый вызов `busMessageFree` логируется с указанием файла и строки:
- `[DataBus] WARN: double-free prevented at protocol_task.cpp:220` — попытка двойного освобождения
- `[DataBus] WARN: busMessageFree unexpected type=5 at kline_task.cpp:70` — сообщение неожиданного типа

## ✅ 8. Чек-лист перед коммитом

- [ ] `publish()` возвращает управление за **≤ 5 мкс** (проверить осциллографом/логгером)
- [ ] Все пакеты передаются через `publishPacket()` с `#pragma pack(1)`
- [ ] JSON-буфер **≥ 384 байт**, `PAYLOAD_MAX ≥ 384`
- [ ] Для `data/*`, `cfg/*`, `status/*` при подписке установлен `retain=true`
- [ ] Фракционная отправка реализована через счётчик пакетов (`counter % 5/10`)
- [ ] Нет `xQueuePeek`, нет указателей на локальный стек, нет `String` в шине
- [ ] При переполнении очереди `FIFO_DROP` инкрементируется `dropCount`, а не крашится задача
- [ ] Каждый `xQueueReceive` завершается вызовом `busMessageFree(&msg)`
- [ ] Нет публикации JSON в `TOPIC_MSG_OUTGOING` кроме Protocol Task
- [ ] K-Line публикует `KlinePack`, Climate публикует `ClimatePack` (не ServicePack!)
